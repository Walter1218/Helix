# Helix 能力缺口分析

## 背景

基于 Helix 定位为**企业级 AI Coding 平台**，分析当前系统已覆盖的能力与潜在缺口，为后续产品规划提供参考。

---

## 一、当前已覆盖的核心能力

### L0 安全层
- **Shadow Worktree**: Git worktree 隔离，提供代码沙箱
- **ToolInterceptor**: AST 级命令过滤，防止危险操作
- **VFSOverlay**: 虚拟文件系统覆盖层

### L1 内存层
- **FTS5 + Vector RAG**: 全文检索 + 语义向量检索
- **Memory Decay**: 内存衰减机制，自动清理过时记忆
- **Multi-LLM Embedding**: 多模型嵌入支持

### L2 控制层
- **Hybrid FSM**: 可挂起/恢复的任务状态机
- **Judge Agent**: 只读对抗节点，防止"偷懒/破坏性"修改
- **AlignmentGuard**: 实时检测偏离目标的行为并自动纠偏
- **TaskGate + Goal**: 任务门控与目标系统

### L3 访问层
- **MCP Server**: Model Context Protocol 服务
- **HTTP API + SSE**: RESTful 接口 + 实时事件流
- **SDK**: JavaScript SDK 供外部集成
- **Event Bus**: 中央事件总线

### 其他能力
- **Actor System**: 多代理协作 (peer/subagent 模式)
- **Workflow Engine**: 内置 deep-research/auto-loop 工作流
- **Auto Dev**: Cron 调度 + Roadmap 集成
- **Spec-driven Development**: 规范驱动开发
- **Trace Reporter**: 执行轨迹记录，支持 DPO 数据集导出

## 二、能力缺口分析

### 2.1 协作层缺失

| 能力 | 说明 | 竞品参考 | 优先级 |
|------|------|----------|--------|
| **多人协作** | 多开发者同时与 AI 交互，冲突检测与合并 | GitHub Copilot Workspace | P1 |
| **会话共享** | 开发者间共享 AI 会话上下文和决策历史 | - | P2 |
| **角色权限** | 不同角色（开发者/审查者/管理员）的差异化权限 | - | P2 |

### 2.2 工程化缺失

| 能力 | 说明 | 竞品参考 | 优先级 |
|------|------|----------|--------|
| **增量构建感知** | 理解项目构建系统，避免全量重编译 | Cursor (部分) | P0 |
| **依赖图分析** | AST 级别的依赖关系追踪，影响范围评估 | - | P1 |
| **测试覆盖率集成** | 实时显示 AI 修改的测试覆盖率变化 | - | P1 |
| **CI/CD 深度集成** | 与 GitHub Actions/GitLab CI 的原生集成 | - | P2 |

### 2.3 智能层缺失

| 能力 | 说明 | 竞品参考 | 优先级 |
|------|------|----------|--------|
| **代码异味检测** | 自动识别技术债务和潜在问题 | SonarQube | P2 |
| **性能分析** | AI 生成代码的性能影响评估 | - | P2 |
| **安全审计** | 自动检测 AI 生成代码的安全漏洞 | Snyk Code | P1 |
| **重构建议** | 主动识别重构机会并提供方案 | - | P3 |

### 2.4 用户体验缺失

| 能力 | 说明 | 竞品参考 | 优先级 |
|------|------|----------|--------|
| **可视化调试** | AI 决策过程的可视化展示 | - | P0 |
| **交互式教学** | AI 解释其决策逻辑，帮助开发者学习 | - | P3 |
| **个性化适配** | 学习开发者编码风格，提供定制化建议 | - | P2 |
| **离线模式** | 本地模型支持，无网络环境可用 | Ollama | P2 |

### 2.5 生态层缺失

| 能力 | 说明 | 竞品参考 | 优先级 |
|------|------|----------|--------|
| **插件市场** | 第三方工具和模型的标准化接入 | VS Code Marketplace | P2 |
| **模型热切换** | 运行时动态切换 AI 模型 | - | P1 |
| **多语言支持** | 针对不同编程语言的专项优化 | - | P2 |
| **IDE 深度集成** | 与 VS Code/JetBrains 的原生插件 | Cursor | P1 |

## 三、建议实施路径

### Phase 1: 工程化基础 (1-2 个月)
- **增量构建感知**: 理解 package.json/scripts，避免重复构建
- **可视化调试**: AI 决策过程的可视化展示
- **依赖图分析**: AST 级依赖追踪

### Phase 2: 智能增强 (2-3 个月)
- **安全审计**: 集成 Snyk/CodeQL
- **模型热切换**: 支持运行时切换模型
- **IDE 集成**: VS Code 插件

### Phase 3: 协作与生态 (3-4 个月)
- **多人协作**: 基于 CRDT 的协同编辑
- **插件市场**: 标准化插件接口
- **个性化适配**: 学习开发者风格

## 四、技术可行性评估

### 高可行性 (可直接复用现有能力)
- 增量构建感知 (复用 AST 分析)
- 依赖图分析 (复用项目探测)
- 可视化调试 (复用 Trace Reporter)

### 中可行性 (需要扩展架构)
- 安全审计 (需集成外部工具)
- 多人协作 (需引入 CRDT)
- 插件市场 (需设计标准化接口)

### 低可行性 (需要重大重构)
- 离线模式 (需重构模型层)
- IDE 深度集成 (需开发原生插件)
