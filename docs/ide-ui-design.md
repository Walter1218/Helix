# Helix VS Code Extension — UI 设计稿 v3

> 基于 `@mimo-ai/ui` 组件库 + Tailwind CSS，适配 VS Code 原生主题。本设计稿在 v2 基础上，新增 Loop Pre-flight 启动前信息收集和 Cardinal 运行时卡点检测，将 Harness 层能力从"被动透明"推进到"主动交互"。

---

## 一、设计原则

1. **原生融合**：继承 VS Code 主题变量（`--vscode-*`），不做自定义皮肤，确保视觉上与编辑器无缝融合。
2. **信息密度优先**：Webview 面板宽度通常 300-500px，比浏览器窗口窄，需压缩横向布局，增加纵向信息密度。
3. **操作即反馈**：所有 Agent 操作（代码修改、文件创建、命令执行）必须在 UI 上给出即时反馈，用户可中断/撤销。
4. **上下文显式化**：当前文件、选中代码、相关文件必须在输入区附近显性展示，让用户感知 Agent 看到了什么。
5. **模式外显化**：**Build / Plan / Compose / Max 四种智能体模式必须在 UI 中显性展示**，用户可主动切换、每条消息标明当前模式、Max 模式实时展示候选进度。模式切换不是配置项，而是高频操作。
6. **Harness 透明化**：权限、记忆、隔离、执行轨迹、成本等 Harness 层能力必须被用户感知，从"能用的 Agent"跃迁为"可信的 Agent"。
7. **启动前确认**：模糊任务在进入 loop 前必须通过 Pre-flight 卡片收集关键信息，用户是"副驾驶"而非"乘客"。
8. **卡点显式化**：运行中检测到卡点时，通过 Cardinal 卡片实时展示暂停原因和可选操作，不让 loop 在黑盒中盲跑。
9. **复用优先**：现有 `packages/app` 的 `message-timeline`、`file-tree`、`prompt-input`、`review-tab` 等组件直接迁移，只调整样式和布局。

## 四、组件详细设计（核心组件）

### 4.1 标题栏 (Panel Header)

**复用**: `SessionHeader` 组件

```
┌────────────────────────────────────────┐
│  ○ 新会话  │ 会话 1  │ 会话 2  +   │  ← 会话标签 (Tab Bar)
├────────────────────────────────────────┤
│  Helix  [🛠️ Build▼] [📋] [🎼] [⚡]  │  ← 模式切换器
│  [模型选择]  ⚙️  🔄  ⛔  🌲  🔒      │  ← 控制按钮 + 状态图标
└────────────────────────────────────────┘
```

**规格**:
- 高度: `36px`
- 背景: `--helix-bg-secondary`
- 边框底部: `1px solid --helix-border`
- 左侧: 品牌 Logo + 模式切换器
- **模式切换器** (Mode Switcher): 紧邻 Logo
  - 按钮组: 四个模式按钮横向排列，当前激活模式高亮
  - 按钮: 高度 `22px`，圆角 `4px`，内边距 `0 8px`，字号 `11px`
  - 颜色: 每个模式有对应色条（`border-left: 3px solid {color}`）
    - Build: `#fb8147` (橙色) · 图标: 🛠️
    - Plan: `#c7e2a8` (浅绿) · 图标: 📋
    - Compose: `#a7a3d8` (浅紫) · 图标: 🎼
    - Max: `#e85d75` (玫红) · 图标: ⚡ (实验性，需配置开启)
  - 激活态: 背景 `--helix-bg-tertiary` + 对应色边框
  - 悬停态: 背景 `--helix-bg-tertiary` + 不透明度 50%
  - 快捷键: `Ctrl+Shift+B` (Build), `Ctrl+Shift+P` (Plan), `Ctrl+Shift+O` (Compose), `Ctrl+Shift+M` (Max)
  - 切换时: 短暂显示切换动画（scale 1.05 → 1），同时清空当前输入框 placeholder 提示
- 右侧: 控制按钮
  - ⚙️ 设置（齿轮图标）
  - 🔄 刷新会话
  - ⛔ 停止生成
  - 🌲 隔离状态（Shadow Worktree）——绿色=主工作区，黄色=隔离中，红色=有未合并修改
  - 🔒 权限状态——绿色=无越界，黄色=有待授权请求，红色=有拒绝操作
- 模型选择: 下拉菜单，显示当前模型和 Provider 图标

│  └────────────────────────────────┘   │
│                                        │

**规格**:
- 消息最大宽度: `100%`（窄面板不需要限制宽度）
- 用户消息: 背景 `--helix-bg-tertiary`，右对齐，无边框
- Agent 消息: 背景 `transparent`，左对齐，带左侧 `2px` 强调色边框
- **模式标识**: 每条 Agent 消息气泡顶部必须显示当前模式标识
  - 格式: `[模式图标] [模式名] · [Agent 名]`，例如 "🛠️ Build · 🤖 Agent"
  - 位置: 消息气泡左上角，字号 `11px`，颜色 `--helix-fg-secondary`
  - 模式色条: 左侧 `2px` 边框颜色跟随模式（Build=橙, Plan=绿, Compose=紫, Max=玫红）
  - 背景: 消息气泡背景根据模式微染色（透明度 5% 的模式主色）
  - Max 模式消息: 额外显示候选进度 "⚡ Max · N/M → {状态}"
  - Compose 模式消息: 额外显示触发 skill 名称的 pill 标签
- **子智能体执行卡片**（新增）：当 `explore`/`general`/`judge` 等 subagent 被调用时
  - 在消息流中插入可折叠的执行卡片
  - 默认折叠，显示简要状态（如 "🔍 正在探索代码库..."）
  - 展开后显示：扫描文件列表、关键发现、进度条
  - Judge 卡片：显示验证目标、检查项列表、通过/失败状态
  - 卡片头部: 子智能体图标 + 名称 + 状态（运行中/已完成/失败）
  - 运行中: 蓝色脉冲圆点 + 旋转图标
  - 完成: 绿色对勾
  - 失败: 红色感叹号
- **DPO 反馈卡片**（新增）：用户点击 👍/👎 后的确认提示
  - 小型 toast 样式，3秒后自动消失
- 代码块: 背景 `--helix-bg-primary`，边框 `1px solid --helix-border`，圆角 `4px`
- 代码块头部: 显示文件路径 + 语言标识 + 复制按钮，高度 `28px`
- Diff 块: 行号 + 变更指示（`+`/`-`）+ 背景色（add/del）
- 操作按钮: 紧凑排列（Accept/Reject/Modify），高度 `24px`，圆角 `4px`
- 工具调用卡片: 显示工具名、参数摘要、执行状态（running/success/error），可展开详情
- **消息操作区**（新增）：每条 Agent 消息底部右侧
  - 👍 标记为高质量 / 👎 标记为低质量（DPO 数据收集）
  - 📋 复制消息内容
  - 🔄 重新生成（仅针对该消息）

**流式响应状态**:
- 思考中: 脉冲动画 + 文字 "Thinking..."（颜色跟随当前模式）
- 工具调用: 展示工具名 + 参数，带旋转图标
- 生成代码: 打字机效果，逐字渲染
- 完成: 脉冲动画停止，显示操作按钮
- 子智能体执行: 显示实时进度条和文件扫描计数

### 4.3 上下文条 (Context Bar)

**新增组件**（`packages/app` 无直接对应，需新建）

```
┌────────────────────────────────────────┐
│  📎 上下文:                             │
│  [src/utils.ts] [L10-25] [x]           │  ← 当前文件 + 选中范围
│  [README.md] [相关] [x]                 │  ← 自动关联文件
│  + 添加文件...                          │  ← 手动添加
└────────────────────────────────────────┘
```

**规格**:
- 高度: `auto`（最小 `32px`，可展开）
- 背景: `--helix-bg-tertiary`
- 边框顶部: `1px solid --helix-border`
- 内边距: `8px 12px`
- 文件标签: 圆角 pill 形状，高度 `22px`，背景 `--helix-bg-input`，带文件图标 + 文件名 + 范围 + 删除按钮
- 自动关联文件: 用虚线边框区分，显示 "auto" 小标签
- 操作: 点击添加文件 → 弹出文件选择器（复用 `dialog-select-file.tsx`）
- 悬停: 显示文件路径 tooltip

**数据源**:
- 当前文件: 自动从 VS Code `window.activeTextEditor` 获取
- 选中范围: 自动从 `selection` 获取，显示行号范围
- 相关文件: 从 `mimo serve` 的 `/project` API 获取推荐文件
- 手动添加: 通过文件树或文件选择器添加

  - `dream`: 显示记忆回顾进度、压缩摘要
  - `distill`: 显示重复工作流识别进度、自动化建议
- 实时更新: 通过 SSE/WebSocket 推送执行状态
- 操作按钮:
  - 终止（⏹）：强制停止子智能体执行
  - 跳过（⏩）：跳过当前子任务，进入下一步
- 完成态: 卡片折叠为简要结果，保留"查看详情"入口

### 5.9 Loop Pre-flight 配置面板 (Pre-flight Checklist)

**目标**：在 loop 启动前收集充分信息，让 Agent 从"盲目猜测"变为"有明确边界的执行"。从"全自动驾驶"转向"用户是副驾驶"——用户设定信任边界，Agent 在边界内自主执行。

```
┌────────────────────────────────────────┐
│  🔍 Pre-flight 检查  ·  Plan 模式       │  ← 头部
├────────────────────────────────────────┤
│  模糊度分析: 0.72 / 1.0 (需确认)         │  ← 模糊度分数
│  ━━━━━━━━━━━━━━━━░░                    │  ← 进度条 (72%)
├────────────────────────────────────────┤
│  📋 需要确认的问题 (3/5):               │  ← 问题列表
│  ┌────────────────────────────────┐   │
│  │ 1. 目标优先级是什么？          │   │
│  │    [性能] [可读性] [安全性]    │   │  ← 快速选择
│  │ 2. 是否允许修改 API 签名？     │   │
│  │    [允许] [不允许] [不确定]    │   │
│  │ 3. 测试通过后直接提交？          │   │
│  │    [是] [否] [仅Diff审查]      │   │
│  └────────────────────────────────┘   │
├────────────────────────────────────────┤
│  📝 补充约束条件                        │  ← 自由输入
│  ┌────────────────────────────────┐   │
│  │ 不修改 xxx.ts 的 exports...    │   │
│  └────────────────────────────────┘   │
├────────────────────────────────────────┤
│  ⚙️ 介入偏好配置                        │  ← 可折叠
│  [✓] Plan 完成后展示摘要再执行          │
│  [✓] 测试失败时暂停询问               │
│  [ ] 修改超过 3 个文件时确认            │
│  [✓] 需要外部依赖时暂停               │
├────────────────────────────────────────┤
│  [确认并启动] [仅启动 (跳过确认)] [取消]│  ← 底部按钮
└────────────────────────────────────────┘
```

**规格**:
- 触发时机:
  - 用户发送消息后，在进入 `runLoop` 之前自动触发（`PreFlightRequired` BusEvent）
  - 仅当模糊度 `score >= threshold`（默认 0.6）时展示
  - 用户可在 `mimocode.json` 中设置 `loop.intervention.preFlight.mode = "skip"` 跳过
- 位置: 消息流中插入为**可交互的 Pre-flight 卡片**（不是弹窗，保留上下文）
- 问题类型:
  - **快速选择**: 预定义选项（如 `[性能] [可读性] [安全性]`），单选/多选
  - **确认开关**: 是/否/询问（如 "是否允许修改 API 签名？"）
  - **自由输入**: 补充约束条件（textarea，可选）
- 问题来源:
  - 轻量模型（`mimo-lite`）根据用户输入和项目上下文自动生成
  - 按模式差异化（Build/Plan/Compose/Max 的问题不同）
  - 用户可自定义默认问题模板（保存在 `mimocode.json`）
- 介入偏好配置:
  - 可折叠区域，展示当前模式的介入偏好（复用 `mimocode.json` 中的 `loop.intervention`）
  - 允许用户在此面板快速修改（仅对当前会话生效）
- 底部操作:
  - **确认并启动**: 提交回答，将回答注入为合成消息，启动 `runLoop`
  - **仅启动 (跳过确认)**: 不等待回答，直接启动（提示"你可能需要确认几个关键点"）
  - **取消**: 取消当前任务，不启动 loop
- 动画: 卡片出现时使用 `translateY 8px→0` + `opacity 0→1`（200ms），问题按序号依次展开（stagger 100ms）
- 无障碍: 每个问题区域有 `aria-label`，Tab 聚焦到选项，Enter 选择

### 5.10 Cardinal 运行时卡点指示器 (Cardinal Detection HUD)

**目标**：在 loop 运行过程中，当检测到卡点时，在 UI 中实时展示结构化暂停，让用户知道 Agent 为什么停下来了，需要用户做什么。

```
┌────────────────────────────────────────┐
│  ⚠️ 检测到卡点: 外部依赖               │  ← 头部（黄色警告）
├────────────────────────────────────────┤
│  类型: external_dep                      │  ← 卡点类型
│  严重度: pause（暂停等待）               │  ← severity
│  详情:                                   │  ← 详情区域
│  Agent 需要安装 `pytest` 以运行测试      │
│  检测命令: npm install --save-dev jest │
│  文件: packages/opencode/package.json  │
├────────────────────────────────────────┤
│  [允许安装] [拒绝安装] [查看上下文]      │  ← 用户操作
│  同时记住此选择 [✓]                      │  ← 持久化选项
└────────────────────────────────────────┘
```

**卡片变体**:

| 卡点类型 | 颜色 | 图标 | 典型操作 |
|---------|------|------|---------|
| **ambiguity** | 🟡 黄色 | ❓ | [确认] [修改目标] [跳过] |
| **external_dep** | 🟡 黄色 | 📦 | [允许安装] [拒绝安装] [查看上下文] |
| **test_failure** | 🔴 红色 | 🧪 | [查看失败详情] [自动修复] [停止] |
| **tool_error** | 🟡 黄色 | ⚠️ | [重试] [忽略] [停止] |
| **token_budget** | 🟠 橙色 | 💰 | [继续] [降格模式] [停止] |
| **heal_exhausted** | 🔴 红色 | 🚫 | [查看修改] [回滚] [停止] |

**规格**:
- 触发时机:
  - `processor.ts` 的 `tool-call` 处理中检测到卡点（`cardinal.ts` 规则引擎）
  - `severity === "pause"` 时插入卡片，等待用户操作
  - `severity === "warn"` 时在状态栏显示警告（不阻塞）
  - `severity === "stop"` 时终止 loop，展示终止摘要
- 位置:
  - 消息流中插入为**可交互的 Cardinal 卡片**（紧跟在触发卡点的消息之后）
  - 状态栏显示实时警告指示器（小圆点 + 文字）
- 卡片结构:
  - 头部: 颜色条（根据 severity）+ 图标 + 卡点类型名
  - 详情区: 自动折叠，展开后显示检测规则、上下文、建议操作
  - 操作区: 根据卡点类型展示不同操作按钮（1-3 个）
  - 持久化选项: "同时记住此选择" 复选框，保存到 `mimocode.json`
- 状态栏指示器:
  - 正常: 无指示
  - `warn`: 黄色脉冲圆点 + 文字 "⚠️ 警告: token 预算 80%"
  - `pause`: 红色闪烁圆点 + 文字 "⏸ 暂停: 等待用户确认"
- 与 Question 工具的整合:
  - 不是替换 `Question` 工具，而是扩展为**模式化的结构化暂停**
  - 当模型调用 `AskUserQuestion` 时，如果检测到是已知卡点类型，自动渲染为 Cardinal 卡片
  - 当模型调用 `Suspend_Task` 时，直接渲染为停止卡片
- 动画: 卡片出现时使用 `scale 0.9→1` + `opacity 0→1`（200ms），紧急程度高的（red）使用脉冲边框
- 无障碍: 严重度用颜色 + 文字 + 图标三重标识，ARIA live region 播报卡点类型

## 六、交互流程

### 6.1 打开 Helix 面板

1. 用户点击活动栏图标（或快捷键 `Ctrl+Shift+H` / `Cmd+Shift+H`）
2. 扩展 Host 创建 Webview Panel，加载 `packages/app` 构建产物
3. Webview 初始化：连接 `mimo serve` WebSocket/SSE，获取当前会话列表和当前模式（默认 Build）
4. 显示默认会话（或创建新会话），标题栏显示当前模式、隔离状态、权限状态

### 6.2 切换模式

1. 用户点击标题栏模式切换器或输入框模式快捷栏，或快捷键 `Ctrl+Shift+B/P/O/M`
2. 当前模式即时切换，UI 反馈：
   - 标题栏高亮新模式的按钮
   - 消息流中的新消息使用新模式的色条
   - 输入框占位符动态切换为新模式的提示语
   - 如果切换到 Compose 模式，文件树面板新增 Skills 标签
   - 如果切换到 Max 模式，标题栏右侧显示 "Max 候选进度" 指示器
   - 如果切换到 Plan 模式，操作按钮变为 "生成计划" 和 "执行计划"（计划可转为 Build 执行）
3. 模式切换不中断当前会话，仅影响下一条消息的交互方式
4. 模式状态持久化到当前 session 的 metadata 中（通过 `mimo serve` API）

### 6.3 发送消息

1. 用户在输入框输入消息，自动收集上下文（当前文件 + 选中代码）
2. 上下文条显示收集到的文件，用户可删除或添加
3. 用户按 `Enter` 发送，消息进入消息流（用户气泡）
4. 消息气泡顶部显示当前模式标识（如 "🛠️ Build"）
5. 显示 "Thinking..." 状态（带模式色脉冲）
6. Agent 开始流式响应，消息气泡实时更新，色条颜色跟随模式
7. 工具调用时显示工具卡片，执行完成后展示结果
8. 代码修改时显示 Diff 预览卡片，等待用户操作
9. Max 模式时：显示候选进度指示器，Judge 完成后高亮最佳候选，用户可切换候选
10. Compose 模式时：显示 skill 触发卡片，展示技能执行流程
11. Plan 模式时：显示计划步骤列表，每个步骤可展开查看详情，底部提供 "执行计划" 按钮（切换为 Build 模式执行）
12. 子智能体执行时：消息流中插入可折叠的执行卡片，展示实时进度
13. 权限请求时：弹出权限确认弹窗，显示权限上下文和策略
14. 响应完成，状态栏更新 Token 用量和预算进度
15. 用户可对 Agent 消息进行 👍/👎 反馈

### 6.4 代码修改操作

1. Agent 返回代码修改（`edit` 工具调用）
2. 消息流中显示 Diff 预览卡片（带当前模式标识，如 "🛠️ Build · 代码修改"）
3. 用户操作:
   - **接受**: 通过 `postMessage` 通知 Host → Host 调用 `applyEdit` 写入文件 → 文件树更新修改状态
   - **拒绝**: 忽略修改，卡片标记为 "已拒绝"（灰色）
   - **Diff**: 在 VS Code 中打开 Diff 编辑器（`vscode.diff`）
   - **修改**: 在编辑器中打开文件，让用户手动调整
4. 批量操作: 在文件树面板中 "全部接受" / "全部拒绝"
5. 如果处于 Shadow Worktree，修改先写入 Worktree，用户确认后再合并到主工作区

### 6.5 行内对话

1. 用户选中代码，按 `Ctrl+K`
2. 弹出 Inline Chat 面板，自动填充选中代码作为上下文
3. 行内对话顶部显示模式选择条（默认继承主面板当前模式，用户可切换）
4. 用户输入指令（如 "解释这段代码"），输入框占位符随当前模式变化
5. 流式响应，展示结果（带模式标识）
6. 用户接受后，修改直接应用到文件（或替换选中代码）

### 6.6 使用 Harness 能力

1. **查看权限**: 用户点击 Side Panel 的 🔒 权限标签 → 查看当前 Agent 权限矩阵 → 点击修改 → 切换 allow/ask/deny → 保存到 mimocode.json
2. **查看记忆**: 用户点击 🧠 记忆标签 → 浏览/搜索记忆 → 删除不需要的记忆 → 标记重要记忆 → 手动添加记忆
3. **查看子智能体执行**: 消息流中展开子智能体执行卡片 → 查看扫描进度 → 发现异常点击终止
4. **查看隔离状态**: 用户点击标题栏 🌲 图标 → 查看 Shadow Worktree 状态 → 选择合并或丢弃
5. **编辑 AGENTS.md**: 用户点击 📋 标签 → 查看/添加/编辑规则 → 修改后提示下条消息生效
6. **管理插件**: 用户点击 🔌 标签 → 查看已加载插件 → 配置/重启/禁用 → 添加新插件
7. **反馈数据质量**: 用户点击 Agent 消息的 👍/👎 → 数据收集 → 累计反馈数显示在状态栏

### 6.7 文件操作

1. 用户点击文件树中的文件
2. 通过 `postMessage` 通知 Host
3. Host 调用 `vscode.window.showTextDocument` 打开文件
4. 文件在 VS Code 编辑器中打开，Webview 保持聚焦（或根据配置切换）

### 6.8 Loop Pre-flight 交互流程

**目标**：在 loop 启动前和运行时，通过 UI 引导用户完成信息收集和卡点决策，将"盲跑 loop"转为"有用户信任的协作 loop"。

#### 6.8.1 启动前预检 (Pre-flight)

```
用户发送消息 → 触发 Pre-flight 检查 → 模糊度分析 → 展示 Pre-flight 卡片
                                      ↓
                              score < 0.6? ──是──→ 直接启动 runLoop
                                      ↓否
                              展示问题列表 (快速选择/确认开关/自由输入)
                                      ↓
                              用户点击 [确认并启动]
                                      ↓
                              回答注入为合成消息 → 启动 runLoop
                                      ↓
                              用户点击 [仅启动]
                                      ↓
                              提示"你可能需要确认几个关键点" → 启动 runLoop
                                      ↓
                              用户点击 [取消]
                                      ↓
                              取消任务，不启动 loop
```

**交互细节**:
1. 用户发送消息后，消息先进入消息流（用户气泡），但状态栏显示 "🔍 分析任务中..."（轻量模型运行中）
2. 分析完成后（约 200-500ms），在消息流中插入 Pre-flight 卡片
3. 如果用户之前设置过默认回答（如"总是允许修改 API 签名"），自动填充并标记为已回答
4. 用户回答问题时，卡片内实时更新进度条（已回答 / 总问题数）
5. 所有问题回答完成后，"确认并启动"按钮变为可用状态（高亮）
6. 用户点击"确认并启动"后，Pre-flight 卡片折叠为摘要（显示已确认的关键点），然后启动 runLoop
7. 如果用户点击"仅启动"，卡片保留在消息流中（灰色，标记为"已跳过"），但 loop 继续运行

#### 6.8.2 运行时卡点处理 (Cardinal Detection)

```
Loop 运行中 → 工具调用 → processor.ts 检测卡点
                                      ↓
                              severity = warn? ──是──→ 状态栏显示警告（不阻塞）
                                      ↓否
                              severity = pause? ──是──→ 插入 Cardinal 卡片，loop 暂停
                                      ↓否
                              severity = stop? ──是──→ 终止 loop，展示终止摘要
                                      ↓否
                              继续运行
```

**交互细节**:
1. 当检测到 `pause` 级别卡点时，loop 暂停，状态栏显示 "⏸ 暂停: 等待用户确认"
2. 在消息流中插入 Cardinal 卡片，紧跟在触发卡点的消息之后
3. 卡片自动展开（动画 200ms），展示详情和可操作按钮
4. 用户操作后:
   - **[允许安装]** → 卡片标记为"已解决"（绿色），loop 继续运行
   - **[拒绝安装]** → 卡片标记为"已拒绝"（灰色），loop 跳过此步骤继续
   - **[修改目标]** → 弹出输入框，用户修改目标后，卡片折叠，loop 以新目标重新规划
   - **[停止]** → 卡片标记为"已停止"（红色），loop 终止，展示终止摘要
5. 如果用户 30 秒未响应，状态栏显示脉冲警告（不自动继续，防止用户不在屏幕前时自动执行）
6. 用户选择"同时记住此选择"时，保存到 `mimocode.json` 的 `loop.intervention.inFlight` 配置中

#### 6.8.3 终止前交付物确认 (Deliverable Gate)

```
Loop 终止前 → taskGate 检查 → goalGate 检查 → deliverableGate 检查
                                      ↓
                              有未确认修改? ──是──→ 插入交付物确认卡片
                                      ↓否
                              正常结束
```

**交互细节**:
1. 当 loop 即将终止时，检查是否有未确认的修改（`edit`/`write`/`multiedit` 的 pending 状态）
2. 如果有，在消息流底部插入交付物确认卡片：
   ```
   ┌────────────────────────────────────────┐
   │  📦 交付物确认                          │
   │  Agent 已完成任务，以下修改等待确认：     │
   │  - edit: src/utils.ts                  │
   │  - write: src/utils.test.ts            │
   │  [查看 Diff] [全部接受] [全部拒绝] [调整] │
   └────────────────────────────────────────┘
   ```
3. 用户点击"查看 Diff" → 在 VS Code 中打开 Diff 编辑器
4. 用户点击"全部接受" → 应用所有修改，loop 正常结束
5. 用户点击"调整" → 用户可逐条修改，loop 等待用户完成
6. 如果用户配置 `postFlight.confirmDeliverable = false`，跳过此步骤

#### 6.8.4 与四种模式的整合

| 模式 | Pre-flight 重点 | Cardinal 常见卡点 | Deliverable 重点 |
|------|----------------|------------------|----------------|
| **Build** | 执行范围、文件修改边界 | 工具错误（权限不足）、连续 read 无 edit | 未确认修改列表 |
| **Plan** | 目标与约束、优先级确认 | 计划过于宏大（步骤>10）、无验证步骤 | 计划可执行性检查 |
| **Compose** | 可用技能、编排策略 | 技能调用失败、循环依赖 | 技能执行结果汇总 |
| **Max** | 评估维度、预算上限 | 候选全失败、Judge 评分过低 | 最佳候选可解释性 |

## 十一、无障碍 (Accessibility)

1. **键盘导航**: 所有交互元素支持 Tab 聚焦，`Enter` 激活，`Escape` 关闭/取消
2. **屏幕阅读器**: 消息流区域标记 `role="log" aria-live="polite"`，流式更新自动播报
3. **对比度**: 所有文字对比度满足 WCAG 2.1 AA（4.5:1），依赖 VS Code 主题保证
4. **焦点指示**: 聚焦元素显示 `outline: 1px solid --helix-accent`
5. **减少动画**: 支持 `prefers-reduced-motion`，关闭所有动画
6. **模式色盲适配**: 模式标识不仅依赖颜色，同时有图标和文字（🛠️/📋/🎼/⚡）
7. **权限色盲适配**: 权限状态不仅依赖颜色，同时有文字标签（允许/询问/拒绝）
8. **ARIA 标签**: 所有工具按钮、模式切换器、权限面板使用明确的 `aria-label`
