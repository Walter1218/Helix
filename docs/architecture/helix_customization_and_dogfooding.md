# Helix: 系统定制化说明与 Dogfooding 指南

本文档总结了 Helix 对原有 MiMo 架构所做的深度定制与基建增强，并为接下来的 Dogfooding（吃狗粮）阶段提供了详细的验证建议。

## 1. Helix 对 MiMo 架构的深度定制化与增强

原有的 MiMo 架构提供了一个强大的对话与工具调用基础，但为了实现“全自动化的 Loop 工程智能体”愿景，Helix 在 L0 到 L3 四个核心层级进行了大量的硬核定制：

### L0 层：绝对安全的沙箱底座
*   **定制点：工作区清道夫 (`WorktreeGC`)**
    *   *问题*：原有系统在异常退出或被强杀时，会留下大量孤儿 `git worktree` 和 `.mimo-lock` 锁文件，导致后续会话资源耗尽。
    *   *Helix 增强*：引入了后台守护的垃圾回收机制，基于 PID 存活检测，在系统启动与运行时自动扫描并强力抹除死进程遗留的脏数据。
*   **定制点：工具防火墙 (`ToolInterceptor` 与高危阻断)**
    *   *Helix 增强*：对 `bash` 工具进行了黑白名单与参数级的拦截，阻断了如 `rm -rf /*` 等破坏性命令，防止自主智能体在“幻觉”中摧毁宿主机。

### L1 层：认知与记忆引擎
*   **定制点：记忆衰减器 (`MemoryDecay`) 与语义签名 (`SemanticHash`)**
    *   *问题*：原有 MiMo 的记忆库（`Auto-Dream` 提取的 `MEMORY.md`）是只增不减的，过期的代码规范会变成误导新会话的“毒药”。
    *   *Helix 增强*：引入了基于 AST 结构契约的 `SemanticHash`。在注入 Prompt 前，`MemoryDecay` 会自动比对记忆关联文件的 Hash。如果文件发生了实质性改变，相关记忆会被自动过滤，防止上下文“毒化”。
*   **定制点：自动项目探针 (`ProjectProbe`)**
    *   *Helix 增强*：在生成 System Prompt 时，探针会自动扫描当前工作区的 `package.json`、`tsconfig.json` 等文件，将“必须使用 pnpm”、“遵循 React 函数式组件”等硬性架构约束自动注入，避免大模型“瞎猜”技术栈。
*   **定制点：动态 AST 依赖图谱 (`AstGraph`)**
    *   *Helix 增强*：引入 `ts-morph` 进行代码契约提取。大模型修改代码前，可通过 `getBlastRadius` 查询“爆炸半径”，并通过 `getContract` 只读取依赖文件的类型定义而非海量实现细节，极大节省 Token 并降低幻觉。

### L2 层：调度与对抗验证
*   **定制点：裁判智能体 (`Judge Agent`) 与对抗机制**
    *   *问题*：原有的大模型在遇到修不好的 Bug 时，倾向于“投机取巧”地删掉测试用例的断言以求通过。
    *   *Helix 增强*：剥夺了执行者修改测试的权限，引入了 `Request_Goal_Revision` 逃生舱工具。执行者必须向具有只读权限的“裁判”申请修改测试，由裁判基于宏观目标进行驳回或批准，形成了坚固的对抗验证闭环。

### L3 层：全链路观测
*   **定制点：链路追踪协议 (`TraceReporter`)**
    *   *Helix 增强*：将原本零散的控制台日志通过内部 Event Bus 统一收集，结构化为包含父子层级、状态、时间戳的多模态追踪树，为前端 UI 绘制“大模型思考图谱”提供直接数据源。

## 2. 接下来的 Dogfooding 阶段建议

Dogfooding（吃狗粮）阶段是检验基建是否坚固的唯一标准。既然底层引擎已经拼装完毕，我们应该用它来开发 Helix 自己的下一个业务需求，或者解决一个真实的复杂 Bug。

### 建议场景一：让 Helix 自己实现一个新工具 (Tool)
**测试目标：自动探针、AST 依赖分析与沙箱自愈。**
*   **任务指令**：“为当前项目开发一个名为 `npm_audit` 的工具，用于扫描并输出项目的安全漏洞。”
*   **观察点**：
    1. 观察 `ProjectProbe` 是否告诉了它当前项目用的是 `bun` 而非 `npm`。
    2. 观察它在修改 `src/tool/index.ts` 注册新工具前，是否用 `AstGraph` 获取了工具注册的契约。
    3. 如果它写错了类型，观察 TypeScript 报错是否能触发它的 `HybridFSM` 自愈循环，并在死循环前被 `Judge Agent` 介入。

### 建议场景二：故意提供一个过期的 `MEMORY.md` 规则
**测试目标：验证记忆衰减机制 (`MemoryDecay`)。**
*   **预置条件**：在 `.mimocode/AGENTS.md` 中人为加入一条指向某个旧文件的错误规则（并附带旧 Hash）。然后修改那个真实文件的逻辑。
*   **任务指令**：“重构上述被修改的文件中的一个函数。”
*   **观察点**：在 `TraceReporter` 输出的日志中，观察 `MemoryDecay` 是否精准拦截了那条过期的规则，并且没有把它放进最终大模型的 System Prompt 中。

### 建议场景三：复杂重构引发“爆炸半径”
**测试目标：测试快照回滚树 (`Snapshot`) 与依赖分析。**
*   **任务指令**：“修改 `src/session/schema.ts` 中的 `SessionID` 类型定义结构。”（这是一个极其底层且被广泛引用的类型）。
*   **观察点**：
    1. 智能体是否因为修改底层类型引发了全盘数百个文件的 TS 报错。
    2. 面对这无法修复的灾难，智能体是否能通过 `Suspend_Task` 或多次自愈失败后，利用底层的 `Snapshot` 引擎一键 `revert`，让工作区瞬间回到重构前的安全状态。

### 执行步骤建议
1.  **启动观测面**：保持前端 TUI / UI 开启，实时监控 `TraceReporter` 抛出的节点。
2.  **输入模糊宏观指令**：不要给微观的“修改 A 文件的 B 行”，直接给出上述建议场景中的宏观业务目标。
3.  **放手不干预**：除了在需要批准高危系统命令时输入 `y`，尽量不要干预它的 FSM 循环，观察它能否自己经历 `Plan -> Do -> 报错 -> RCA 分析 -> 恢复快照/修改测试 -> 成功` 的完整双螺旋闭环。
